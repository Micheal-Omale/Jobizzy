**11 Filter + Sort + Pagination (2026-06-25)**

- **All filtering/sorting is client-side, derived in `useJobs`.** `refresh()` already loads the user's full job list (RLS-scoped, newest-first), so the match filter, text search, and sort run as a single `filteredJobs` computed over that in-memory list rather than re-querying the DB per control change. Job counts are small (≤10 per search) so this is instant and avoids a round-trip on every keystroke/dropdown.
- **Filter/sort state lives in `useJobs` (`query`/`matchFilter`/`sort` as `useState`), not a new composable.** `Filters.vue` `v-model`s them; `Table.vue` reads `filteredJobs` and paginates. This keeps the lift-state-up decision from the restore note — the two sibling components share state without prop drilling or events.
- **Filter semantics:** High = `match_score >= MATCH_THRESHOLD` (70, from `lib/utils.ts` — never hardcoded), Low = `< MATCH_THRESHOLD`, with a missing `match_score` treated as 0 (never a high match). Text search is case-insensitive over `company` + `title`. Sort: `score` (desc, default), `newest`/`oldest` by `found_at`. Pagination stays at 20/page (already real from feature 10) and resets to page 1 on any filter/sort/search change.
- **Two distinct empty states:** "No jobs yet" (user has zero jobs → run a search) vs "No matching jobs" (jobs exist but filtered out → adjust filters), so a filter that hides everything doesn't read like the search failed.

**06 Profile Save Logic — post-review fixes (2026-06-24)**

Three bugs found via `/review` + live repro after the first build, all fixed:

- **🔴 `years_experience` crash (the real "saving doesn't work").** `<input type="number">` + `v-model` makes Vue coerce the bound ref to a **number** once a value is entered (empty stays `""`) — Vue's `vModelText` casts when `type === "number"`, no `.number` modifier needed. `buildPayload` called `yearsExperience.value.trim()`, which threw `TypeError: trim is not a function` the moment Years of Experience was filled. The synchronous throw inside `handleSubmit` skipped `saving.value = false`, so the button stuck on **"Saving… forever"** (this — not the résumé upload — was the earlier hang). Fix: `yearsExperience` is `ref<string | number>`; `buildPayload` coerces both (`typeof === "number" ? n : parseInt`). **Lesson: my first repro passed a numeric `years_experience` straight into the payload and never exercised `buildPayload`, so it missed the bug — always repro through the real client builder.**
- **🔴 Missing-profile-row resilience.** `onehmike@gmail.com` predates the signup trigger → no profile row; `saveProfile` only `UPDATE`d → could never save. Fix: `fetchProfile` uses **`maybeSingle()`** (missing row loads as an empty profile, no error) and `saveProfile` uses **`upsert({ id, email, ...update, is_complete })`** (creates the row on first save, scoped by RLS to `auth.uid()`). Verified live: upsert on an existing row updates, on a missing row inserts; maybeSingle returns `data: null` with no error. The one-time SQL backfill was blocked by the safety classifier and is **unnecessary** — upsert self-heals pre-trigger users on their first save.
- **🟡→🔴 Résumé upload was broken, now fixed (3 sub-bugs).** Investigation (live): InsForge uploads via a **presigned POST straight to AWS S3** (`insforge-storage.s3.us-east-2.amazonaws.com`), **~12.5s/MB** measured — a 3-5MB résumé takes ~40-60s. **06 Profile Save Logic**

- **Writes go through the authenticated browser client, NOT a Nitro route — a deliberate, approved deviation from the architecture invariant** ("server-side writes use createInsforgeServer / Composable → Nitro API → DB write"). The only alternatives were (a) client-side writes or (b) migrate feature-02 auth to `@insforge/sdk/ssr` createBrowserClient + a local refresh router now — user chose (a). **04 Database Schema**

- **Schema applied via the InsForge `run-raw-sql` MCP tool** (no migration-file mechanism exists). Four `public`-schema tables created exactly per `architecture.md`: `profiles` (24 cols), `agent_runs` (8), `jobs` (23), `agent_logs` (7). All verified: RLS enabled, 1 policy each, both triggers live, `resumes` bucket private.
- **RLS is real Postgres RLS keyed on `auth.uid()`.** InsForge ships Supabase-style helpers (`auth.uid()` uuid, `auth.role()`, `auth.jwt()`) and the standard `anon` / `authenticated` / `postgres` / `project_admin` roles. Each table has one `own_rows` policy `FOR ALL TO authenticated USING/WITH CHECK` — `profiles` on `id = auth.uid()`, the other three on `user_id = auth.uid()`. `SELECT/INSERT/UPDATE/DELETE` granted to `authenticated` only (no `anon` — all data is user-scoped). The server client carries the user JWT, so agent writes in Nitro routes are correctly scoped with **no service-role bypass**.
- **`auth.users.id` is `uuid`** (default `gen_random_uuid()`) — so `profiles.id uuid REFERENCES auth.users(id) ON DELETE CASCADE` is correct as written in `architecture.md`. `agent_runs`/`jobs`/`agent_logs.user_id` → `profiles(id) ON DELETE CASCADE`; `jobs.run_id` and `agent_logs.job_id` use `ON DELETE SET NULL`; `agent_logs.run_id` cascades.
- **Profiles auto-created on signup via a DB trigger.** `public.handle_new_user()` (SECURITY DEFINER, `search_path=public`) inserts `(id, email)` from `auth.users` `ON CONFLICT DO NOTHING`, fired by `trg_auth_user_created AFTER INSERT ON auth.users`. Every user always has a profile row — **feature 06 only UPDATEs, never inserts**. (The `auth.users` trigger created without ownership issues under the admin SQL role.)
- **Schema hardened beyond the bare column list:** CHECK constraints on `jobs.source ('search','url')`, `agent_runs.status`, `agent_logs.level`, and the profile enums (`experience_level`/`remote_preference`/`cover_letter_tone`/`work_authorization`) — all NULL-tolerant so the auto-created empty profile and partial saves stay valid. `jobs.match_score` CHECK `0–100` (nullable). `job_type` left as free text (Adzuna values vary — a CHECK would break agent inserts). Indexes on `user_id` / `run_id` / `found_at DESC` / `match_score DESC` (+ composite `jobs(user_id, found_at DESC)`) for the feature 11/15/16 filter/sort/stat queries. `trg_profiles_updated_at` (`public.set_updated_at()`) bumps `profiles.updated_at` on UPDATE.
- **Resumes bucket is private (`isPublic: false`), own-files enforced in app code** — the `create-bucket` MCP tool only exposes public/private, so isolation lives in the server route that always builds `resumes/{user_id}/resume.pdf` from the session user (never a client-supplied path). Storage-level RLS on `storage.objects` was deferred (InsForge honoring of it is unverified).
- **Row types written to `types/index.ts`** (root-level, per the `architecture.md` folder layout) — `Profile`, `AgentRun`, `Job`, `AgentLog` plus the enum unions and the `WorkExperienceEntry` / `Education` / `CompanyResearch` jsonb shapes. Single source of truth for table shapes; nullable DB columns typed as `T | null`. Import from the rootDir alias (`~~/types`), since Nuxt 4 srcDir is `app/` so `~`/`@` resolve to `app/`.

**02 Auth**

- **InsForge package is `@insforge/sdk`, not `@insforge/ssr`.** The live MCP docs (which outrank the context files per `library-docs.md`'s order of authority) use `@insforge/sdk` with `createClient`. Browser auth goes through `app/composables/useInsforge.ts` (`createClient`). - **Auth state** lives in `useState` (`auth:user`, `auth:loaded`) via `app/composables/useAuth.ts`; `ensureLoaded()` memoizes the first `getCurrentUser()` call so the middleware doesn't refetch per navigation.

## Notes

- **OAuth redirect URL / port gotcha (debugged 2026-06-23):** The InsForge backend validates the OAuth `redirect_uri` against its allowed redirect URLs. `http://localhost:3000/...` IS accepted out of the box (despite `get-backend-metadata` reporting `allowedRedirectUrls: []` — the backend default-allows localhost:3000). **Any other port is rejected** with `"<url> is not in the allowed redirect URLs"`. Symptom seen: leftover dev servers squatted on :3000, so `nuxt dev` drifted to **:3001**, and `signInWithOAuth` failed at init with a 400. Fix: **always run the dev server on port 3000** (kill stray node listeners on 3000–3002 first). For the deployed app (or any non-3000 dev port), add that exact callback URL to `allowedRedirectUrls` in the InsForge dashboard — no MCP tool sets this.
- **OAuth `Request timed out after 30000ms` (debugged 2026-06-24):** `auth.signInWithOAuth` does one `GET /api/auth/oauth/{provider}` that relays through InsForge's shared auth gateway (`api.insforge.dev`) to mint the provider URL. That call is **slow and variable** — measured 0.8s–8.5s, and on a cold backend it exceeds the SDK's **default 30s timeout** (`InsForgeConfig.timeout`, default 30000), firing intermittently. Backend, CORS (`ACAO: http://localhost:3000`), providers, and the redirect allowlist are all fine — it's pure latency. **Fix:** `useAuth.signInWithOAuth` now uses a dedicated client via `useInsforge({ timeout: 60_000 })` with `skipBrowserRedirect: true`, then redirects manually with `window.location.href = data.url`. Only the handshake gets the longer timeout; the cached default client stays 30s for DB/storage/session reads. Safe — PKCE verifier lives in shared `sessionStorage` (`insforge_pkce_verifier`), so the callback's default client still completes the exchange. Still must run dev on **port 3000** (the only allowed redirect URL).
